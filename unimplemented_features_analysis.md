Ranking (Easiest → Hardest)
Tier 1: Trivial No-Op
1. where clauses — Already parsed as AST_NODE_WHERE_CLAUSE attached to ProcedureSignature. No polymorphic procedures exist yet (item 8), so where has nothing to constrain. Literally just needs case AST_NODE_WHERE_CLAUSE: break; in sem and IR to skip it. 5 minutes of work.
Tier 2: Easy (infrastructure exists, limited scope)
2. #assert directive — Already parsed as AST_NODE_DIRECTIVE_WITH_ARGS. Needs compile-time expression evaluation to validate the condition. The infrastructure for parsing is complete; just need to evaluate the expression at sem time and emit an error if false.
3. Variadic parameters (..) — is_variadic flag is already parsed and passed to LLVMFunctionType (which supports varargs). The grammar, AST, and type system plumbing are complete. Need to: (a) register variadic parameters in IR gen, (b) handle variadic argument forwarding at call sites. No va_* intrinsics needed for simple extern FFI.
4. foreign blocks / imports — Already parsed with library name and nested declarations. The IR generator already has ir_gen_top_level_decl for processing declarations. Foreign declarations just need to be emitted as LLVM declare external function declarations. The semantic analyser needs to walk the foreign block and register the library linkage. Well-defined and bounded.
5. Top-level using — Already parsed as AST_NODE_USING_DECL. Useful for bringing names from another package/scope into the current namespace. The struct-field using embedding already works (type_descriptors.c:731-753), giving a reference implementation pattern. The scope module already has scope_merge primitives.
Tier 3: Moderate
6. when declaration — Already parsed with full grammar+AST. Can reuse the when statement pattern (treat as runtime if at top level, llvm_ir_generator.c:3963). Harder than statement-when because it needs to handle top-level declarations inside conditional branches and register them in the symbol table.
7. any type / RTTI — The packing/unpacking infrastructure is already built (variable init, assignment, type assertion all work). The only missing piece is type IDs: type_id is hardcoded to 0 everywhere (llvm_ir_generator.c:883, 1693). Need to: (a) assign unique integers to type descriptors, (b) store the type ID when packing into any, (c) compare it when unpacking. No structural changes needed — the {i8*, i64} struct is right, the field is just always 0.
8. Multiple return values — Grammar + AST + actions fully wired. ProcMetadata already has return_count and returns[]. But get_or_create_proc_type explicitly ignores them (type_descriptors.c:478-479, (void)returns;). Need to: (a) plumb return list through proc type creation, (b) emit LLVM struct returns, (c) handle destructuring at call sites. Moderate because it touches sem, type descriptors, and IR gen.
9. auto_cast — Keyword reserved, not used in any grammar rule. Would add implicit numeric conversion rules to the semantic analyser (sem_evaluate_expr). The rules are well-defined in the language spec (integer widening, float/int promotion). This is a semantic-only change — the IR generator already handles the right LLVM cast opcodes.
10. #load directive — Needs file I/O at compile time. Grammar already handles it as AST_NODE_DIRECTIVE_WITH_ARGS. Would read a file into a string constant at compile time. Requires: (a) file reading during semantic analysis, (b) injecting the result as a string literal AST node, (c) handling expression context correctly.
Tier 4: Complex
11. union type — Grammar + AST + TD_KIND_UNION + is_type_node all in place. Needs: (a) get_or_create_union_type in type_descriptors.c (tagged union: {i64 tag, [union of fields]}), (b) semantic analyser to resolve fields and create the type, (c) IR generator to allocate with tag+payload and generate member accesses. The struct_members.h infrastructure (struct_or_union_members_st) can be reused.
12. Type assertion x.(T) for unions — Currently only works for any. For unions it needs: (a) the union type implemented (item 11), (b) runtime tag comparison against the requested variant, (c) field extraction from the union payload. Directly depends on item 11.
Tier 5: Very Complex
13. context built-in — Depends on implicit context parameter (item 14). Needs: (a) context type definition (allocator, logger struct), (b) context identifier resolving to the injected parameter, (c) field access on context.
14. Implicit context parameter — Architecture-wide change: inject ^context as hidden first parameter to every procedure, thread it through every call site. Zero existing infrastructure.
15. Inline asm — Keyword reserved, no grammar rule or AST node exists. Needs: (a) grammar rule for asm { ... }, (b) parsing of constraints (output/input/clobber), (c) LLVMConstInlineAsm integration. Well-bounded but significant new code.
16. soa layout (Structure of Arrays) — Grammar + AST + is_type_node all in place. But this is a major memory layout optimization affecting: type creation, member access GEP computation, array/slice indexing through SOA fields, and struct type representation. No existing infrastructure.
Tier 6: Architecture-Altering
17. Polymorphic procedures / monomorphisation — Full generics system. Complete template-like type parameter substitution, procedure instantiation with concrete types, duplicating LLVM functions per unique instantiation, caching. Zero infrastructure exists beyond the AST_NODE_POLY_IDENT parser token. By far the largest and most complex item.
